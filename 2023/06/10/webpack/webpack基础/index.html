初识 webpack

本质上，webpack 是一个用于现代 JavaScript 应用程序的 _静态模块打包工具_。当 webpack 处理应用程序时，它会在内部从一个或多个入口点构建一个 依赖图(dependency graph)，然后将你项目中所需的每一个模块组合成一个或多个 _bundles_，它们均为静态资源，用于展示你的内容

这里需要提到开发环境与生产环境的差异：

开发：

  1. 需要模块化、帮助更好的开发
  2. 会使用一些 ES 新语法和框架特殊写法

生产：

  1. 浏览器自身无法解析模块化
  2. 浏览器只认识 html、css、js，甚至有些浏览器不识别新的 js 语法

而 webpack 作为构建工具，核心工作就是为了解决开发环境与生产环境的矛盾

那么构建工具到底帮我们做了什么？

  1. 编译浏览器无法理解的东西 —— ES6、TS、Vue 语法等（配合插件、loader）
  2. 代替一些人工操作—— 文件合并和拆分 、 图片压缩 、资源处理等
  3. 辅助开发 —— 开发模式

资源文件的处理

在 webpack 5 之前，通常使用：

​ row-loader将文件导入为字符串

​ url-loader将文件作为 data URI 内联到 bundle 中(base64)

​ file-loader将文件发送到输出目录

更多细节参考：https://www.webpackjs.com/guides/asset-modules/

而 webpack5 则自带了对资源文件的支持：

  • asset/resource 发送一个单独的文件并导出 URL。之前通过使用 file-loader 实现。
  • asset/inline 导出一个资源的 data URI。之前通过使用 url-loader 实现。
  • asset/source 导出资源的源代码。之前通过使用 raw-loader 实现。
  • asset 在导出一个 data URI 和发送一个单独的文件之间自动选择。之前通过使用 url-loader，并且配置资源体积限制实现。

html 的处理

html 做什么？

  1. 给 webpack 提供一个 html 模板给，复用固定内容
  2. 打包的时候再生成一个 html
  3. 打包出来的 html 自动引入 js

对于 html 的处理使用插件、而不是loader

为什么不用 loader?

这里需要在加强一下 loader 的概念，当非 js 的代码引入到 js 文件，需要使用 loader 去帮助 webpack 识别， 而 html 文件几乎是不会存在引入 js 的情况。
